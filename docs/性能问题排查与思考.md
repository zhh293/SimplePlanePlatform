# 项目性能问题排查：发现、思考与解决

> 本文记录项目开发过程中两次关键性能问题的完整排查过程。
> 重点不在于最终解决方案本身，而在于**如何从一个异常现象出发，逐步推导出根本原因**。

---

## 问题一：数据传输吞吐极低，页面完全无法加载

### 1. 发现问题

系统联通之后，在浏览器里访问任意网页，页面始终加载不出来。初步判断不是连接问题，因为 CONNECT 握手是成功的，代理通道建立了，但数据就是传不过来。

### 2. 定位异常

回头审视整个数据流的设计：

最初的设计思路是用 **Future 一对一** 的模式处理数据传输——客户端发一条消息，服务端回一条响应，Future 等到响应之后 complete，再把数据写回浏览器。

这套模型在 CONNECT 握手阶段是完全正确的：发一个 CONNECT，等一个 CONNECT_RESPONSE，有来有回。

但数据传输阶段出了问题。

### 3. 推导根因

思考数据实际是怎么在 HTTP/2 上传输的：

一条 ProxyMessage 如果携带的数据较大（比如一个 TLS Record 可以有 16KB），它在被封装成 HTTP/2 DATA 帧发送时，**会被拆成多个帧**，因为 HTTP/2 单帧有最大长度限制（默认 16384 字节）。

而原来的 Decoder 是这样处理的：

```
收到一个 DATA 帧 → 直接解码 → 传到上层
```

这意味着：

- 第一帧到达 → Decoder 尝试解码 → 数据不完整，但还是触发了 Future complete
- 后续的帧到达 → 没有人处理了，因为 Future 已经结束

结果就是：**Future 被一个残缺的数据片段提前结束，剩余的数据帧全部丢失，浏览器什么都收不到。**

本质上是把**流式传输**强行套进了**请求-响应**模型，两者根本不匹配。

### 4. 解决思路

既然 DATA 帧天然是流式的，就不应该用 Future 来处理它。

重新设计数据面的传输模型：

- **控制面（CONNECT / DISCONNECT）**：保留 Future 模型，有 requestId，有超时，有重试。这类消息确实是一问一答的。
- **数据面（DATA）**：彻底放弃 Future，requestId 固定为 0，发后即忘。服务端收到目标响应后，通过 streamId 找到对应的浏览器通道，直接回推数据。

同时在 Decoder 层引入**累积缓冲区**：收到每个 DATA 帧时不立即解码，而是先追加到缓冲区，等缓冲区里积累的字节数达到一条完整 ProxyMessage 的长度之后，再提取并传递给上层。这样无论消息被拆成多少帧，都能完整还原。

### 5. 结果

页面可以正常打开了。

## 问题二：小页面快，大页面慢——GitHub 要加载好几秒

### 1. 发现问题

解决第一个问题之后，基本功能跑通了，但体验上有明显异常：

- `curl` 访问 Google 首页，响应很快，几乎秒出。
- 用浏览器打开 GitHub，要等好几秒才能完整加载出来。

这两个现象放在一起，第一反应不是"代理出了 bug"，而是**两者之间有一个关键变量不同**。

### 2. 设计对照实验

Google 首页体积很小（十几 KB），GitHub 页面加上 JS、CSS 资源有好几 MB。

唯一显著的差异就是**响应体的大小**。

由此形成第一个假设：

> 响应资源超过某个大小阈值之后，传输性能会急剧下降，甚至出现阶段性停顿。

### 3. 推导根因

要理解这个问题，先得搞清楚数据从 proxy-remote 到 proxy-local 这条路径上，到底有哪些流控闸门在起作用。

**3.1 两层窗口模型**

proxy-remote 向 proxy-local 发数据，路径上存在两级独立的流控机制：

**TCP 层：** 底层的 TCP 连接由内核管理。接收方（proxy-local 所在机器）的接收缓冲区初始 128KB（macOS 默认 `recvspace = 131072`），内核通过 auto-tuning 可以动态扩大到 4MB。发送方的拥塞窗口（cwnd）从约 10 个 MSS（~14KB）开始慢启动，每经过一个 RTT 翻倍。**TCP 层在连接刚建立时窗口较小，但会自动增长，稳态下不是瓶颈。**

**HTTP/2 层：** 在 TCP 之上，HTTP/2 还有自己的一套流控窗口，且分为两级——**Stream 级窗口**（每条流独立）和 **Connection 级窗口**（整条连接共享，所有流叠加计算）。RFC 7540 规定两者的默认初始值都是 **65535 字节**。

关键区别在于：TCP 窗口由内核自动调优会越来越大，但 HTTP/2 窗口**不会自动增长**——它的大小完全取决于接收方是否及时发送 WINDOW_UPDATE 帧归还额度。如果不主动扩大，它就永远是 65535 字节。

**3.2 访问 Google 时的窗口状态**

`curl` 请求 Google 首页，只有一条 stream，响应体约十几 KB：

```
Stream 级窗口: 65535 bytes → 消耗 ~15KB → 剩余 ~50KB（充裕）
Connection 级窗口: 65535 bytes → 消耗 ~15KB → 剩余 ~50KB（充裕）
TCP cwnd: 慢启动阶段约 14KB~56KB → 足够传完（1~2 个 RTT）
```

数据量远小于任何一层窗口的容量，全程没有触发限流，一气呵成。

**3.3 访问 GitHub 时的窗口状态**

浏览器打开 GitHub，同时发出几十个并发请求（主 HTML + JS + CSS + 图片……），全部复用同一条 HTTP/2 连接。此时：

**Stream 级窗口：** 每条流各自有 65535 字节，单个资源只要不超过 64KB 就不触发 stream 级限流。但 GitHub 的 JS bundle 动辄上百 KB，单条流也会触发多次等待。

**Connection 级窗口（关键瓶颈）：** 所有流共享 65535 字节的连接级配额。假设有 30 条流同时活跃，平均每条流只能分到约 2KB 就触发连接级限流。**连接级窗口耗尽后，全部流同时阻塞**，必须等接收方发回 WINDOW_UPDATE 才能继续。

**TCP cwnd：** 如果是复用的长连接，cwnd 已经在高位（几百 KB 甚至 MB 级别），TCP 层本身不是瓶颈。但如果是新连接，TCP 慢启动初期 cwnd 很小（~14KB），会和 HTTP/2 窗口形成**双重瓶颈叠加**——TCP 刚给了一点数据，HTTP/2 窗口就满了；HTTP/2 窗口释放了一点，TCP cwnd 还没涨上去又限制了发送。

**3.4 停等循环的量化分析**

假设跨洋链路 RTT ≈ 300ms，连接级窗口 = 65535 字节：

```
30 条流同时请求 → Connection 窗口瞬间耗尽 → 所有流阻塞
→ 等待 WINDOW_UPDATE 回来（1 个 RTT ≈ 300ms）
→ Netty 收到数据后自动消费并发回 WINDOW_UPDATE（窗口归还 ≈ 65KB）
→ 30 条流竞争这 65KB → 每条流分到 ~2KB → 立即再次耗尽
→ 再等 300ms → ……
```

这就是经典的**带宽时延积（BDP）瓶颈**：

```
有效吞吐上限 = 窗口大小 ÷ RTT
Connection 级: 65535 ÷ 0.3s ≈ 213 KB/s
```

213 KB/s 的吞吐上限，传输 GitHub 几 MB 的资源，需要好几秒——和实际观察完全吻合。

而 Google 首页 ~15KB 的数据量，在第一个窗口周期（不需要任何 WINDOW_UPDATE 往返）内就传完了，根本感受不到这个限制。

**3.5 为什么问题出在 HTTP/2 层而不是 TCP 层？**

TCP 有自动调优机制——内核检测到 BDP 较大时会主动扩大接收缓冲区和通告窗口，稳态下 TCP rwnd 可以增长到 MB 级别。但 HTTP/2 的窗口初始值是协议写死的 65535，**不会自动增长**，只能通过 SETTINGS 帧（设置 stream 级初始窗口）或手动发送 WINDOW_UPDATE 来扩大。Netty 的默认行为是"消费多少归还多少"——它发的 WINDOW_UPDATE 只是归还已消费的字节，不会把窗口撑到比初始值更大。

所以真正卡脖子的是 HTTP/2 连接级窗口这道闸门：TCP 那边有足够的带宽和窗口可以传数据，但 HTTP/2 层不允许发——发送方的 HTTP/2 连接级可用窗口为 0，只能干等。

### 4. 解决思路

根因是 HTTP/2 连接级窗口太小，与高 RTT 链路的 BDP 严重不匹配。解决方案：**在连接建立后立即通过 WINDOW_UPDATE 帧将连接级窗口扩大到足够覆盖 BDP 的水平**。

HTTP/2 协议允许通过 WINDOW_UPDATE 帧将窗口累加到最大 2^31 - 1 字节。将连接级窗口从 65535 扩大到 **16MB**，同时将 stream 级初始窗口通过 SETTINGS 帧设为 **1MB**：

```
扩大后的连接级吞吐上限:
16MB ÷ 300ms ≈ 53 MB/s

扩大后的单 stream 吞吐上限:
1MB ÷ 300ms ≈ 3.3 MB/s
```

53 MB/s 远超实际链路带宽，HTTP/2 窗口不再是瓶颈。同时 stream 级 1MB 也足够保证单个大资源（如 JS bundle）不会频繁停等。

窗口扩大后，限流的角色自然回归到 TCP 层：由内核的 cwnd（拥塞控制）和 rwnd（接收缓冲区）来根据实际网络状况动态调节速率。

**TCP 慢启动阶段的实际过程（新建连接，RTT ≈ 300ms）：**

新连接建立时，TCP 的 cwnd 从约 10 MSS（≈14.6KB）起步。HTTP/2 窗口已有 16MB，完全不构成限制，此时 TCP 慢启动是唯一的节流点，每过一个 RTT cwnd 翻倍：

```
t =    0ms：cwnd ≈  14KB  → 请求刚发出，浏览器白屏
t =  300ms：cwnd ≈  28KB  → 首批数据到达，HTML 开始解析
t =  600ms：cwnd ≈  56KB  → CSS / 首屏 JS 开始下载，页面骨架出现
t =  900ms：cwnd ≈ 112KB  → 多路资源并发到达，页面逐步可见
t = 1200ms：cwnd ≈ 224KB
t = 1500ms：cwnd ≈ 448KB
t = 1800ms：cwnd ≈ 896KB  → 绝大多数资源已到达
t ≈ 2100ms：cwnd ≈ 1.8MB  → 进入稳态，TCP 本身不再是瓶颈
```

**用户体感是"顺畅的渐进式加载"**：页面从空白开始逐渐有内容，2–3 秒内完全可交互。和修复前的本质区别在于：HTTP/2 冻结–解冻循环消失了，TCP 慢启动是平滑增长，每个 RTT 都能看到进展，不会有大段的无响应停顿。

对于**长连接复用**的场景（浏览器保持代理连接不断开），cwnd 早已在高位，慢启动阶段完全不存在，体感接近秒开。

这正是合理的分工——**TCP 层做拥塞控制和端到端流控，HTTP/2 层只做多路复用的公平调度，不应该成为吞吐瓶颈**。

### 5. 结果

GitHub 页面加载恢复到正常速度，不再出现阶段性卡顿。

## 两个问题的共同思路

回顾这两次排查，有一个共同的模式：

**第一步：不急着改代码，先问"为什么会这样"。**

遇到现象之后，第一反应是找出**变量**——是什么东西不同，导致了两种截然不同的结果。跨帧问题的变量是"消息是否被拆帧"，窗口问题的变量是"响应体的大小"。找到变量，就找到了排查的方向。

**第二步：从现象推机制，而不是从代码找 bug。**

两次都没有去翻代码一行一行找哪里写错了，而是先在脑子里把数据流跑一遍，想清楚"数据应该怎么流"和"数据实际在哪里卡住了"之间的差距。差距就是根因。

**第三步：解决方案只针对根因，不打补丁。**

跨帧问题的根因是"模型不匹配"，解决方案是换模型，而不是在原来的 Future 上打各种补丁。窗口问题的根因是"配额不够"，解决方案是扩配额，而不是去优化数据压缩或者减少并发数。

对症下药，改动最小，效果最直接。
